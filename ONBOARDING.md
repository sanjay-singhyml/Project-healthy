## 1. Project Overview

`fuckray` is a **TypeScript‑based CLI/Server toolkit** that:

* Analyses a repository for CI/CD health, code quality, documentation coverage, flakiness, security, PR complexity, environment configuration and build performance.  
* Leverages **AI** (via the `proxy` package) to generate actionable suggestions, fixes and reports.  
* Provides a rich set of commands (`ci-check`, `config-wizard`, `diff`, `explore`, `fix`, `trend`, …) that can be run locally or in CI pipelines.  
* Exposes a lightweight HTTP **AI‑proxy** server that can be used by other tools (e.g., VS Code extensions) to fetch AI‑generated fixes on‑the‑fly.

The repo is split into two logical parts:

| Area | Purpose |
|------|---------|
| `src/cli` | The command‑line interface (parses arguments, orchestrates modules). |
| `src/modules/*` | Individual “analysis modules” (`m01‑cicd`, `m02‑quality`, …) each returning a standardized result shape. |
| `src/proxy` | Minimal Express server that forwards requests to an LLM provider (OpenAI, Anthropic, …) and implements RAG (retrieval‑augmented generation). |
| `src/utils` | Shared helpers (logging, GitHub fetching, output formatting, etc.). |
| `tests/` | Vitest test‑suite covering all public APIs and CLI behaviours. |

## 2. Tech Stack

| Layer | Technology | Why it’s used |
|-------|------------|---------------|
| Runtime | **Node.js ≥ 18** (LTS) | Modern ES modules, native fetch, good performance. |
| Language | **TypeScript 4.x** | Strong typing, excellent IDE support. |
| Build | **tsc** (via npm scripts) → `dist/` | Simple compile‑to‑JS pipeline; no bundler needed. |
| CLI | **commander** (via `src/cli/index.ts`) | Declarative command definition, sub‑commands, auto‑help. |
| Server | **express** (in `src/proxy/server.ts`) | Tiny HTTP API for AI proxy. |
| HTTP client | **node‑fetch** (or native fetch) | Calls LLM providers and GitHub APIs. |
| Testing | **Vitest** (`vitest.config.ts`) | Fast, Vite‑compatible test runner with built‑in mocking. |
| Lint/Format | **ESLint** + **Prettier** (via `package.json` scripts) | Consistent code style. |
| CI | **GitHub Actions** (referenced in `m01‑cicd`) | Automated testing, linting, release. |
| Documentation | **Typedoc** (optional) | Generates API docs from TypeScript. |

## 3. Key Files & Entry Points

| Path | Role |
|------|------|
| `src/index.ts` | **Main entry point** – exports the public API (`run`, `orchestrator`, CLI bootstrap). |
| `src/cli/index.ts` | CLI bootstrap – registers all commands (`ci-check`, `explore`, …) and hands control to the orchestrator. |
| `src/orchestrator.ts` | Core orchestrator – loads enabled modules, merges results, and formats output. |
| `src/modules/*/index.ts` | Individual analysis modules (CICD, quality, docs, …). Each implements `run(context): Promise<ModuleResult>`. |
| `src/proxy/server.ts` | Express server exposing `/chat`, `/rag` endpoints for AI interactions. |
| `src/proxy/ai-client.ts` | Thin wrapper around the LLM provider (OpenAI/Anthropic). |
| `src/utils/output.ts` | **Largest file** – formats results into SARIF, JSON, table, or markdown. |
| `src/config.ts` | Central configuration loader (environment variables, `.fuckrayrc`, defaults). |
| `tests/**/*.test.ts` | Test suite – run with `npm test`. |
| `bin/ph.js` | Executable shim (generated by `npm link`/`pnpm link`). |
| `dist/` | Compiled JavaScript output (used by the published npm package). |
| `package.json` | Scripts, dependencies, `bin` field (`"fuckray": "./dist/index.js"`). |
| `vitest.config.ts` | Vitest configuration (coverage, globals). |

> **Quick start** (after cloning):  
> ```bash
> npm ci            # install exact deps
> npm run build    # compile src → dist
> npm link          # makes `fuckray` available globally
> fuckray --help    # see the CLI
> ```

## 4. Architecture Overview

```
+-------------------+          +-------------------+
|   CLI (commander) |  ---->   |  Orchestrator     |
+-------------------+          +-------------------+
          |                           |
          | loads                     | loads
          v                           v
+-------------------+   +-------------------+   +-------------------+
| Module m01‑cicd   |   | Module m02‑quality|   | Module m03‑docs   |
+-------------------+   +-------------------+   +-------------------+
          |                           |                |
          |   …  (m04‑flakiness, …)   |                |
          +---------------------------+----------------+
                               |
                               v
                     +-------------------+
                     |  Output Formatter |
                     |  (src/utils/out)  |
                     +-------------------+
                               |
                               v
                      +-----------------+
                      |  Console / File |
                      +-----------------+

Separate side‑car:
+-------------------+    +-------------------+
|  AI Proxy Server  |←→  |  AI Client (LLM) |
+-------------------+    +-------------------+
```

### Core Concepts

| Concept | Description |
|---------|-------------|
| **Context** (`src/context/index.ts`) | Holds repo‑wide data (git root, changed files, config, cache). Passed to every module. |
| **ModuleResult** (`src/types/index.ts`) | Standard shape: `{ name, status, data, suggestions? }`. Enables unified reporting. |
| **Cache** (`src/cache/*`) | AST and docs builders store heavy artefacts to speed up repeated runs. |
| **RAG** (`src/proxy/rag.ts`) | Retrieves relevant repo files, injects them into LLM prompts for more accurate fixes. |
| **Fix Strategies** (`src/fix/strategies.ts`) | Defines how AI‑generated patches are applied (inline diff, file overwrite, etc.). |
| **Logger** (`src/utils/logger.ts`) | Centralised structured logging (JSON or pretty). |
| **Output Formats** (`src/formatters/sarif.ts`, `src/utils/output.ts`) | SARIF (for CI), markdown, JSON, or console tables. |

## 5. Development Workflow

| Step | Command | What it does |
|------|---------|--------------|
| **Install** | `npm ci` | Exact versions from `package-lock.json`. |
| **Compile** | `npm run build` | Runs `tsc` → `dist/`. |
| **Lint** | `npm run lint` | ESLint + Prettier check. |
| **Test** | `npm test` (or `npm run test:watch`) | Runs Vitest. |
| **Run CLI locally** | `node dist/index.js <cmd>` or `fuckray <cmd>` (after `npm link`) | Execute any command (e.g., `fuckray ci-check`). |
| **Run AI proxy** | `node dist/proxy/server.js` | Starts the local AI proxy on `localhost:3000`. |
| **Add a new module** | `src/modules/mXX-<name>/index.ts` | Export a `run(context)` function returning `ModuleResult`. Register it in `src/orchestrator.ts`. |
| **Commit** | `git commit -m "feat: …"` | Follow Conventional Commits. |
| **Push & PR** | `git push && open PR` | CI runs tests + lint; reviewers check coverage. |

### Branching Model

* **`main`** – always production‑ready (published to npm).  
* **Feature branches** – `feat/<short-name>` (e.g., `feat/m09‑ai‑proxy`).  
* **Bugfix branches** – `fix/<short-name>`.  
* **Release branches** – `release/vX.Y.Z` (only for version bump).  

All PRs must pass the **GitHub Actions** workflow (`npm ci && npm run lint && npm test && npm run build`).

## 6. Testing

* **Framework** – Vitest (compatible with Vite, fast, built‑in mocking).  
* **Location** – `tests/` mirrors the source tree (e.g., `tests/m02-quality.test.ts`).  
* **Run** – `npm test` (runs once) or `npm run test:watch` (watch mode).  
* **Coverage** – `npm run test:coverage` generates an HTML report in `coverage/`.  

### Writing a Test

```ts
import { describe, it, expect } from 'vitest';
import { run as qualityRun } from '../src/modules/m02-quality';

describe('quality module', () => {
  it('detects missing docs', async () => {
    const ctx = await createFakeContext({ /* … */ });
    const result = await qualityRun(ctx);
    expect(result.status).toBe('warning');
    expect(result.suggestions).toContain('Add JSDoc comments');
  });
});
```

*All new code must have at least 80 % branch coverage.*

## 7. Common Issues & Tips

| Symptom | Likely Cause | Fix |
|---------|--------------|-----|
| **`Cannot find module '@src/...`** | `tsconfig` paths not resolved in IDE or test runner. | Ensure `npm run build` (or `ts-node`) is run, and add `moduleNameMapper` in `vitest.config.ts` if using custom aliases. |
| **`dist/index.js` missing after `npm start`** | Forget to run `npm run build`. | Run `npm run build` first, or use `npm run dev` (watch mode) which builds on the fly. |
| **AI proxy returns 401** | LLM API key not set. | Export `OPENAI_API_KEY` (or `ANTHROPIC_API_KEY`) in your shell or add to `.env`. |
| **Tests flaky on CI** | `src/utils/github-fetcher.ts` hitting rate limits. | Mock network calls (`vi.mock(...)`) or use the cached fixtures in `tests/fixtures/`. |
| **`npm install` hangs on Windows** | Long path names (especially under `node_modules`). | Use Node 18+ with `corepack` (pnpm) or enable long path support (`git config core.longpaths true`). |
| **`npm run lint` fails on Prettier** | Inconsistent line endings (`CRLF` vs `LF`). | Run `git config --global core.autocrlf input` and re‑format files (`npm run format`). |
| **`src/utils/output.ts` TypeScript errors** | This file is large and occasionally out‑of‑sync after refactors. | Run `npm run lint -- --fix` and then `npm run build`. If the error persists, open an issue – it’s a known hot‑spot. |
| **CLI crashes with “ENOENT: no such file or directory, open …/package.json”** | Running the binary from a directory that isn’t the repo root. | Always execute `fuckray` from the repository root or set `FUCKRAY_ROOT` env var. |

### Tips for New Contributors

1. **Start small** – add a unit test or a tiny new command (`src/cli/commands/hello.ts`).  
2. **Run the full pipeline locally** – `npm run lint && npm test && npm run build && fuckray health`.  
3. **Read the module pattern** – look at `src/modules/m01-cicd/index.ts` for a canonical implementation.  
4. **Leverage the cache** – heavy operations (AST parsing, Git diff) are cached under `src/cache`. Re‑use `Cache` utils to avoid recomputation.  
5. **Ask for help** – the core maintainers (`sanjay-shahyml`, `sanjay-singhyml`) are responsive on the project’s Slack channel.
